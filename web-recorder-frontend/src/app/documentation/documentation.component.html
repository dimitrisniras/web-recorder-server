Getting Started with Web Recorder

Web Recorder is an automated website testing and monitoring service that checks for problems with your website or application. It carries out operations in a browser, the same way a user would, to ensure that everything is working properly.


In order to start recording your own test cases, firstly you should sign up for a free Web Recorder account. For test creation you can either install our Chrome extension or you can use our hosted test editor. Tests are run from our cloud service. You can set tests to run continuously on a daily schedule and control various other aspects of your testing.


You can learn more about Web Recorder and its capabilities in the next sections

Chrome Extension

Web Recorder includes a free Chrome extension for recording actions in your browser and turning them into a reproducible test case. This extension is available in the GitHub. After installing the extension, a Web Recorder icon will appear in your toolbar. Clicking this icon will show you the Web Recorder recording tool. The first time that you will access the extension, you will need to log in to your Web Recorder account. Then a recording button will present to you.



The extension works by recording the actions you perform in the browser, such as clicking, filling forms, select values etc. so that those actions can be formed into a test and the same sequence can be run again from the Web Recorder cloud service. Once you click "Start Recording" the extension will begin recording your actions.


In addition to recording actions, the extension can also record "Assertions". Assertions are confirmations that something is true, for example, that an element exists on the page. They are helpful to verify that your test is working as it should. During the recording process, you can switch from recording actions to recording assertions by clicking the Web Recorder toolbar icon and the "Make Assertions" button.



When recording assertions, the extension will attempt to disable any actions associated with elements so that you can assert things like links and buttons without triggering their associated actions. You'll see the extension displays a crosshair cursor and highlights elements as you hover over them so you can tell what you're asserting. These assertions can be adjusted in the test editor after recording is complete. You can change between recording actions and assertions at any time during the recording process. There is also a small link to cancel the recording and start over.


Once you're done recording, click the "Done" button. You'll be presented with a form for saving the test. You will need to name the test and select which of your suites to store it in.



Once you save the recording, you'll be presented with a link to the new test in the hosted Web Recorder application. Once the test case upload to Web Recorder application, you can edit the test steps.


Test Steps

Web Recorder's test editor provides you with the ability to manage all the steps in your test. It is accessible by clicking the "Edit Steps" button when viewing your test. Tests are often created with our Chrome extension first and then been edited. However, tests can also be built from scratch. The test editor displays each test step as a block with options for adding and removing steps.



The dropdown menu on the bottom, left contains the action associated with the step. This dictates whether the step is performing an operation such as clicking an element, making an assertion, and so on. There is a wide range of actions available.



When a step involves interacting with an element on the page, for instance clicking a button, our system targets that element using its id, name, text, class, CSS or XPath. The selector for the target element is entered into the "Identifier" input at the top of the step. Also next to actions dropdown menu there is another one that gives you the ability to choose what type of targeting you want.


In the left corner there is a link "Apply to all", that implements any change that have been made in the step's identifier in all tests that have the same element and URL. Down from it there is an input that you can type any description about the specific step.



Certain actions will present additional fields for the step. For instance, an "Assign" step will present a field for inputing the value to assign to the element you are targeting. Also a drowdown menu will apear with values "Free Text", "Regular Expression", "Integer Range" and "Float range". These options can modify the input value with a text that user enters, a regular expression or an integer or float in a scepcific range. A "Keypress" step will allow you to choose from a number of keystrokes like "Enter", "Backspace" and "Escape". Each step action has it's own set of fields related to the action it's performing.



Furthermore there is a search input that you can type the step that you want to find. The keywords are based on the identity of the step and it's description.




Step Actions

Operations

Operations include the standard actions that a typical user will carry out on a webpage.


Click

Execute a mouse left click on a specific element.

Double Click

Execute a mouse double click on a specific element.

Right Click

Execute a mouse right click on a specific element.

Assign

Assign a value into an input field, such as an <input type="text"> element.

Select

Assign a value into an input field, such as an <select> element.

Keypress

Perform a keypress action, such as pressing "Enter" on an element.

Mouse Over

Move the mouse cursor over an element, triggering "mouseover" events.

Pause

Pause for a specific amount of time. The time is specific in milliseconds, so a value of 1000 will pause the test for 1 second.


Navigation

Navigation actions include two single operations for navigating to a page and refreshing it.


Go to URL

Navigate to a specified URL in the browser.

Refresh

Refresh the current page in the browser.


Assertions

Assertions allow you to confirm that a certain condition is met on the page. They help to confirm that things are working as expected.

Element is present

This test will pass if the specified element is present in the DOM. It will fail if it is not.

Element is not present

This test will pass if the specified element is not present in the DOM. It will fail if it is.

Element is visible

This test will pass if the specified element is visible on the page. It will fail if the element is not visible.

Element is not visible

This test will pass if the specified element is not visible on the page. It will fail if the element is visible.

Element text equals

This test will pass if the specified element's text (or value) is exactly equal to the included text. It will fail if the element's text does not exactly match.

Element text does not equal

This test will pass if the specified element's text (or value) is not equal to the included text. It will fail if the element's text does exactly match.

Element text contains

This test will pass if the specified element's text (or value) contains the included text. It will fail if the element's text does not contain the text.

Element text does not contain

This test will pass if the specified element's text (or value) does not contain the included text. It will fail if the element's text does contain the text.

Modularizing Tests

Web Recorder gives you the ability to include preconditions and postconditions to your tests. That means that you can execute, before (preconditons) or after (postconditions) the specified test, another test (or tests) that you had created before. This allows you to segment out certain operations and easily reuse them.

Scheduling Suite Runs

Web Recorder gives you the ability to execute your tests in a specified suite on a daily schedule. To use this feature, go into your suite's settings and modify schedule.

API Documentation

Web Recorder is a free and open source project. You can find the code of the server here and the code of the Chrome extension here. Our API allows you to access and run tests remotely. It can be used to integrate Web Recorder into your workflow.


In order to access the API you have to be authorized. For that reason on API calls (except Authenticate, Register, Forgot Password and Reset Password calls) you have to use authorization headers with this format type: Authorization: Bearer + <user.token> (ex. Authorization: Bearer xxxxxxx). The user's token can be found following these steps: Open DevTools > Application > Local Storage > currentUser > token. API calls also need user's unique ID. This can be found exactly like user's token.

Security


Accessing Data

Web Recorder uses best practices for Internet security. This helps ensure that your data is safe, secure, and available only to authorized users. Your data will be completely inaccessible to anyone else, unless you explicitly choose to share that data with the public. Web Recorder enforces secure HTTPS for our entire website. All communications with Web Recorder's API are also protected with SSL.


The Chrome extension allows you to record your actions and save them in a test case. It only does this when you manually start recording a test. It’s not tracking your actions otherwise. The recording stops the second you finish (or cancel). Furthermore, the extension only sends the recorded data to our servers when you save the test. It does not send this data to our servers during the recording process.


Storing Data

Web Recorder allows you to store browser actions in a test and execute them from our servers. This can include logging into websites with a set of credentials. All test information is stored as plain text in our database and does not be encrypted. While we take the most precautions to secure our servers and database , if a database breach were to occur, test data would be accessible. For this reason, we insist that you never use live credentials or any type of sensitive data within a test, whether specified manually or recorded with our extension.
